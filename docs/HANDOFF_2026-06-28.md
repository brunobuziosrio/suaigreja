### 3. Opt-out rastreavel do WhatsApp

Implementado localmente e deployado na VPS:

- Migration nova:
  - `supabase/migrations/20260628120000_whatsapp_opt_outs.sql`
- Lib nova:
  - `src/lib/whatsapp-consent.server.ts`
- Endpoint novo:
  - `src/routes/api.public.whatsapp-opt-out.ts`
- `src/routeTree.gen.ts` atualizado pelo build para incluir `/api/public/whatsapp-opt-out`.
- `docs/ROADMAP_IMPLEMENTACAO.md` atualizado com a entrega.

Comportamento implementado:

- Nova tabela `whatsapp_opt_outs`.
- RLS/admin policies para opt-out.
- Funcao `normalize_whatsapp_phone(text)`.
- Funcao `record_whatsapp_opt_out(...)`.
- `claim_whatsapp_messages(integer, integer)` agora retorna tambem `member_id`.
- Mensagens automaticas recebem o aviso:
  - `Responda SAIR para cancelar.`
- Enfileiramento manual de WhatsApp bloqueia destinatario que ja fez opt-out.
- Cron de aniversarios pula opt-out e contabiliza `skippedOptOut`.
- Cron de dizimo pula opt-out e contabiliza `skippedOptOut`.
- Dispatcher verifica opt-out antes do envio.
- Se houver opt-out no dispatcher:
  - reembolsa credito com idempotency key `refund:opt_out:{message.id}`;
  - marca mensagem como `skipped`;
  - limpa `locked_until`;
  - registra erro amigavel: `Destinatario retirou o consentimento para WhatsApp.`

Backup antes da migration de opt-out:

- `/opt/igreja/backups/pre_whatsapp_optout_20260628_012448.dump`

Observacao sobre migration:

- A primeira tentativa de aplicar a migration falhou ao trocar o retorno de `claim_whatsapp_messages`.
- A migration foi corrigida para executar `DROP FUNCTION IF EXISTS public.claim_whatsapp_messages(integer, integer);` antes de recriar a funcao.
- Depois disso, a migration aplicou com sucesso.
